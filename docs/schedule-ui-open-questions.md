# スケジュール設定フロントエンド 要件不足事項一覧

作成日: 2026-07-09
更新日: 2026-07-18
対象: [概要設計書](./schedule-ui-overview-design.md)

対話形式でのヒアリングにより No.1〜12 すべて決定済み。

## 決定済み事項

| No. | 論点 | 決定内容 |
| --- | --- | --- |
| 1 | 編集対象の範囲 | holiday を含む hour/minutes(ON/OFF)まで編集対象とする。`minute_settings` は編集対象外とし、保存時も破壊せず温存する |
| 2 | 保存のタイミング | 明示的な保存ボタン方式(トグル即時保存はしない) |
| 3 | 利用者とアクセス範囲 | 家庭内 LAN・単一利用者を想定 |
| 4 | 認証・認可の要否 | なし(No.3 が LAN 限定・単一利用者のため) |
| 5 | 同時編集の扱い | 楽観ロックは導入せず、後勝ち(最後の保存が有効) |
| 6 | 手編集との併用 | 併用しない。UI を唯一の編集手段とする(`schedules.json` の直接編集は運用上想定しない) |
| 7 | 公開ポート・URL | `3000` を直接公開、HTTP のみ(外部公開ではないため HTTPS 不要) |
| 9 | 初期化方針 | GET 時にファイルが無い/バリデーションエラーの場合は `initialized: false` を返し、UI 側で「空の週間スケジュールで始める」か「サンプル設定(`sample_schedules.json`)からコピーして始める」かを選択させる |
| 10 | 分ボタンの粒度 | 5 分刻み(0/5/10/…/55 の 12 ボタン)で選択できるようにする |
| 11 | 保存履歴・バックアップの要否 | 保存前に `schedules.json.bak` を 1 世代分残す(上書き、世代管理はしない) |
| 8 | コンテナとホストのファイル権限(`settings/` の共有方式) | ホストへの bind mount はせず、Docker の named volume を frontend/backend 両コンテナで共有する。volume の作成・所有権管理は backend 側リポジトリ(別リポジトリ)の責務とし、このリポジトリは `external: true` で参照するのみ。ホスト側 cron 実行ユーザーの UID/GID を事前確認する必要はなくなった |
| 12 | フロント用コンテナの起動管理 | `restart: unless-stopped` のみで運用する。`docker.service` の自動起動に任せ、自前の systemd unit は用意しない(`docker compose down` 後は改めて `up -d` が必要) |
