Next.js 特有の注意点

説明

Next.js の App Router には、ブラウザの標準的な React アプリと違う点がいくつかある。 テストを書く前に、この3点を押さえておかないと「なぜか動かない/なぜかテストできない」に直面する。

1. jsdom 環境が必要

コンポーネントテストは実ブラウザではなく Node.js 上で疑似 DOM(jsdom)を使って動かす。 vitest.config.mts で environment: 'jsdom' を指定する。

2. next/navigation のモックが必要な場合がある

useRouter/usePathname/useSearchParams を使うコンポーネントは、 テスト実行環境に実際のルーターが存在しないためモックが必要になることがある。

3. 非同期 Server Component はユニットテスト不可

Next.js 公式ドキュメントに明記されている制約:「async Server Components は React エコシステムでまだ新しく、 Vitest は現時点でサポートしていない」。同期コンポーネントはユニットテストできるが、 非同期コンポーネントは E2E テスト(Playwright) で検証する方針にする。

コード例(Vitest 設定)

// vitest.config.mts
import { defineConfig } from 'vitest/config'
import react from '@vitejs/plugin-react'
import tsconfigPaths from 'vite-tsconfig-paths'

export default defineConfig({
  plugins: [tsconfigPaths(), react()],
  test: {
    environment: 'jsdom',
  },
})

// next/navigation のモック例
import { vi } from 'vitest'

vi.mock('next/navigation', () => ({
  useRouter: () => ({ push: vi.fn(), replace: vi.fn() }),
  usePathname: () => '/',
}))

ポイント

  • コンポーネントテストは environment: 'jsdom' が前提
  • next/navigation 系フックを使う画面は、テスト側でモックする
  • 非同期 Server Component(データ取得を伴う画面)は Vitest 対象外 → Playwright の担当
  • 本プロジェクトのスケジュール画面はクライアントコンポーネント中心の想定のため、 大部分は Vitest + RTL でカバーできる見込み

確認問題

  1. コンポーネントテストで environment: 'jsdom' が必要な理由を説明してください。
  2. 非同期 Server Component の検証を Vitest ではなく Playwright に任せるのはなぜですか?
