Concept 03 / 07

React Testing Library でのコンポーネントテスト

説明

React Testing Library(RTL)の中心思想は 「実際の使われ方に近いテストほど、テストは信頼できる」というもの。 コンポーネントの内部実装(state名や関数名)ではなく、 画面に見えているもの・ユーザーが押せるものを基準にテストを書く。

例えるなら、店員の接客チェックを「マニュアル通りの内部手順」で採点するのではなく、 「お客さんから見てちゃんと案内できているか」で採点するイメージ。 実装をリファクタリングしても、見た目の振る舞いが変わらなければテストは壊れない。

コード例

// app/page.tsx
export default function Page() {
  return (
    <div>
      <h1>Home</h1>
      <button>保存</button>
    </div>
  )
}

// __tests__/page.test.tsx
import { expect, test } from 'vitest'
import { render, screen } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import Page from '../app/page'

test('見出しが表示される', () => {
  render(<Page />)
  expect(screen.getByRole('heading', { level: 1, name: 'Home' })).toBeDefined()
})

test('保存ボタンを押せる', async () => {
  const user = userEvent.setup()
  render(<Page />)
  await user.click(screen.getByRole('button', { name: '保存' }))
})

ポイント

  • render でコンポーネントを描画し、screen で画面上の要素を取得する
  • 要素の取得は「ロール名+アクセシブルな名前」(getByRole)を優先する。 `data-testid` は他に手段がない場合の最終手段
  • userEvent は実際のユーザー操作(クリック・入力)に近い形でイベントを発火する
  • 本プロジェクトのスケジュール画面では「時刻ボタンのトグル」「保存ボタン押下」を この形式でテストする想定

確認問題

  1. RTL が「実装の詳細」ではなく何を基準にテストすべきと考えているか説明してください。
  2. getByRole と data-testid はどちらを優先して使うべきですか? 理由も含めて答えてください。
