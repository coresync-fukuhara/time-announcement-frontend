Concept 06 / 07

MSW によるAPIモック

説明

MSW(Mock Service Worker)は、リクエストをネットワークレベルでインターセプトしてモックのレスポンスを返す仕組み。 「どの HTTP クライアントを使っているか(fetch か axios か)」をテストコードが気にする必要がない点が最大の利点。

例えるなら、コンビニの本部(外部API)に発注をかける代わりに、 店の前に「本部そっくりの受付窓口」を用意して、本当に発注が飛んだかのように振る舞わせるイメージ。 アプリ側のコードは相手が本物かモックかを区別できない。

本プロジェクトでは BFF がファイル I/O(fs)を行うため外部 HTTP 通信は多くないが、 将来外部 API(通知連携など)を追加した場合や、フロントエンドのフェッチ処理をテストする場合に有効。

コード例

// mocks/handlers.ts
import { http, HttpResponse } from 'msw'

export const handlers = [
  http.get('/api/schedules', () => {
    return HttpResponse.json({
      monday: [{ hour: 9, minutes: [0, 30] }],
    })
  }),
]

// mocks/server.ts (Node.js/テスト用)
import { setupServer } from 'msw/node'
import { handlers } from './handlers'

export const server = setupServer(...handlers)

// vitest.setup.ts
import { beforeAll, afterEach, afterAll } from 'vitest'
import { server } from './mocks/server'

beforeAll(() => server.listen())
afterEach(() => server.resetHandlers())
afterAll(() => server.close())

ポイント

  • http.get/http.put など Web 標準の Request/Response に近い形で書ける
  • setupServer(Node.js用)と setupWorker(ブラウザ用)があり、同じ handlers を使い回せる
  • beforeAll/afterEach/afterAll でモックサーバーの起動・リセット・終了を行うのが定型パターン
  • 本プロジェクトの現段階(外部API通信なし)では必須ではないが、 フロントエンドの fetch('/api/schedules') 呼び出しをコンポーネントテストで検証する際に活用できる

確認問題

  1. MSW が「ライブラリ単位」ではなく「ネットワークレベル」でインターセプトすることの利点を説明してください。
  2. afterEach で server.resetHandlers() を呼ぶ理由を考えて説明してください。
