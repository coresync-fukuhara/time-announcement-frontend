Playwright による E2E テスト

説明

Playwright は実際のブラウザ(Chromium/Firefox/WebKit)を自動操作して、 「ユーザーがブラウザで実際に行う操作」を丸ごと検証するツール。 ユニットテストでは検証できない、非同期 Server Component を含む画面や、 複数の操作をまたぐシナリオ(タブ切り替え→トグル操作→保存→再読み込みして確認)はここで担保する。

例えるなら、部品検査(ユニットテスト)と工程検査(APIテスト)を終えた後に行う 「実際にお客さんとして使ってみる試運転」。手間はかかるが、実際に動くかどうかの最終確認になる。

コード例

// e2e/schedule-editing.spec.ts
import { test, expect } from '@playwright/test'

test('月曜9:00をONにして保存すると再読み込み後も反映されている', async ({ page }) => {
  await page.goto('/')

  await page.getByRole('tab', { name: '月' }).click()
  await page.getByRole('button', { name: '9時 00分' }).click()
  await page.getByRole('button', { name: '保存' }).click()

  await expect(page.getByText('保存しました')).toBeVisible()

  await page.reload()
  await expect(page.getByRole('button', { name: '9時 00分' })).toHaveAttribute(
    'aria-pressed',
    'true'
  )
})

# セットアップ(初回のみ)
npm init playwright@latest

# 実行
npx playwright test

# ブラウザを表示しながら実行(デバッグ用)
npx playwright test --headed

ポイント

  • page.goto/page.getByRole/expect(locator).toBeVisible() など 「Web-first assertions」を使い、要素が現れるまで自動で待つ
  • 並列実行・複数ブラウザエンジン対応・トレースビューアが無料(Cypressは並列実行が有料)
  • 本プロジェクトでは「初期化ダイアログの選択」「曜日タブ切り替え」「保存→再読み込み後の反映確認」 など画面をまたぐ主要シナリオをE2Eの対象にする
  • ユニット/APIテストで担保できる範囲はE2Eに持ち込まない(重複させて実行時間を無駄にしない)

確認問題

  1. ユニットテストではなくE2Eテストでしか確認できないことの例を1つ挙げてください。
  2. Playwright が Cypress に対して持つ、CIコスト面での優位性を説明してください。
