Concept 01 / 07

全体像:なぜこの組み合わせなのか

説明

テストには大きく3つの層がある。ユニット/コンポーネント(1つの関数・1つの画面部品)、 API(BFF)(Route Handlers 1本の入出力)、E2E(ブラウザを実際に動かして画面から画面までの流れ)。 層ごとに得意なツールが違うので、1つのツールで全部をカバーしようとしない。

例えるなら、ユニットテストは「部品単体の検品」、API テストは「工程ごとの検品」、 E2E は「完成品の試運転」。試運転だけで全部を確認しようとすると遅くて壊れやすい試験になる。

比較表(2025〜2026年の実態)

層採用不採用/補助根拠
ユニット/コンポーネント Vitest + React Testing Library Jest(満足度低下傾向)、node:test(機能が薄い) Next.js公式ドキュメントがVitestのセットアップ手順を掲載。ESM/TypeScriptがそのまま動き、Jest比で大幅に高速
API(Route Handlers) Vitest + next-test-api-route-handler(NTARH) スーパーテスト系ライブラリ(Express前提で非採用) NextRequest/NextResponseをそのまま扱えるNext.js専用ヘルパー
外部APIモック MSW(Mock Service Worker) nock(Node.js限定・legacy向け) ネットワークレベルでインターセプトするため、ブラウザ/Node両方・fetch APIでもそのまま使える
E2E Playwright Cypress(並列実行が有料、非同期Server Componentの確認はE2Eに委ねる必要がある) State of JS 2025で「最も採用が伸びたツール」。CI費用も低く、Next.js公式でも紹介

コード例(全体構成イメージ)

frontend/
├── vitest.config.mts        # Vitest設定(jsdom環境)
├── playwright.config.ts     # Playwright設定
├── __tests__/               # ユニット/コンポーネントテスト
│   └── page.test.tsx
├── app/api/schedules/
│   ├── route.ts
│   └── route.test.ts        # NTARHでのRoute Handlerテスト
└── e2e/
    └── schedule-editing.spec.ts  # Playwright E2E

ポイント

  • 「全部をE2Eで確認する」は遅く壊れやすい。ピラミッド型(ユニット多め・E2E少なめ)を意識する
  • 非同期 Server Component は Vitest では検証できない制約がある → その部分だけ Playwright(E2E)で担保する
  • TDD では「まず失敗するテストを書く」ため、実行速度が速いVitestの採用が特に効いてくる

確認問題

  1. ユニットテスト・APIテスト・E2Eテストは、それぞれ何を検証する層か説明してください。
  2. なぜ非同期 Server Component は Vitest だけで検証できないのか説明してください。
