Route Handlers(BFF)のテスト

説明

本プロジェクトの BFF(app/api/schedules/route.ts)は Next.js の Route Handler として実装される。 Route Handler は NextRequest を受け取り Response を返す関数だが、 ブラウザや実サーバーを起動せずにテストしたい。そこで使うのが next-test-api-route-handler(NTARH)。

例えるなら、実際の郵便システム(HTTPサーバー)を丸ごと動かさなくても、 「窓口の人(ハンドラー関数)に直接封筒(リクエスト)を渡して、返ってきた封筒(レスポンス)を確認する」ような検査方法。

コード例

// app/api/schedules/route.test.ts
import { testApiHandler } from 'next-test-api-route-handler' // 最初にimportする
import { describe, it, expect, vi } from 'vitest'
import * as appHandler from './route'
import * as store from '@/lib/schedule-store'

describe('GET /api/schedules', () => {
  it('保存済みのスケジュールを返す', async () => {
    vi.spyOn(store, 'readSchedules').mockResolvedValue({
      monday: [{ hour: 9, minutes: [0] }],
      /* ...省略... */
    })

    await testApiHandler({
      appHandler,
      async test({ fetch }) {
        const res = await fetch({ method: 'GET' })
        expect(res.status).toBe(200)
        const body = await res.json()
        expect(body.monday[0].hour).toBe(9)
      },
    })
  })
})

describe('PUT /api/schedules', () => {
  it('バリデーション違反なら400を返す', async () => {
    await testApiHandler({
      appHandler,
      async test({ fetch }) {
        const res = await fetch({
          method: 'PUT',
          body: JSON.stringify({ invalid: true }),
        })
        expect(res.status).toBe(400)
      },
    })
  })
})

ポイント

  • testApiHandler の import は他の import より前に書く(Next.js の内部動作の都合)
  • appHandler にテスト対象の route.ts をそのまま渡す
  • vi.mock(完全に差し替え)と vi.spyOn(元の実装を保ちつつ呼び出しを監視)を使い分ける。 ファイル I/O 部分(schedule-store.ts)は spyOn/mock で差し替え、実ファイルには触れないようにする
  • テストは「実行(fetchでリクエスト送信)→ 検証(ステータス・ボディ)→ 確認(モックが正しく呼ばれたか)」の順で書く

確認問題

  1. NTARH を使うと、実際のHTTPサーバーを起動せずに何をテストできますか?
  2. vi.mock と vi.spyOn の違いを説明してください。
