「package manager まで必要か」を判断する手順

01で挙げた目安を、実際に順番に当てはめると次のようになる。

  1. Step1: 現状の運用で何に困っているか — 困りごとが特にない(1人で使っていて、都度 .mcp.json を手で書けば済む)なら、 02〜05すら不要で現状維持でよい
  2. Step2: 困りごとが「発見しにくい」「隔離実行したい」「マーケットプレイスで配りたい」なら → 02〜05(発見・インストール系)で足りる。追加の依存管理・ロックファイルは過剰
  3. Step3: 困りごとが「環境間で構成がズレる」「チーム/CIで寸分違わず再現したい」 「Skillだけでなく全部丸ごとロック・統制したい」なら → 06 Microsoft APM のような宣言的依存管理まで踏み込む価値がある
  4. Step4: 「Skill流通の実績・コミュニティの厚みだけを重視したい」なら → 07 Vercel Skills で十分。フルスタックな依存管理(APM)は過剰な場合が多い

このプロジェクト(schedule-ui)への当てはめ

本リポジトリは家庭内LAN・単一利用者運用が前提で、認証も楽観ロックも持たない設計になっている (schedule-ui-overview-design.md)。 Step1〜4に照らすと、「複数人でのチーム開発」「CI環境での再現性」「組織的なガバナンス」は いずれも現時点では当てはまらない。したがって Microsoft APM のような package manager 級の ツールを今すぐ導入する必要性は低く、MCP サーバーを増やす場面が来たとしても、 02〜05の発見・インストール系(特に軽量な install-mcp や、Docker前提ならDocker MCP Toolkit)で 十分間に合う可能性が高い。将来チーム開発やCI連携が発生した時点で、改めて06〜08を見直すとよい。

判断の切り口(その他の軸)

  • Skill配布も含めて統一管理したい(単一クライアント前提) → Claude Code Plugins
  • 複数クライアントを横断して使う/チームの標準を揃えたい(MCPのみ) → Smithery か Docker MCP Toolkit
  • 信頼性の低い/出所不明なMCPサーバーも試したい、隔離実行を重視したい → Docker MCP Toolkit
  • とにかく手早く1つだけ追加したい、余計な依存を増やしたくない → install-mcp (本プロジェクトの .mcp.json への手動追記も実質これに近い方式)
  • Skill/MCP/Hookをまとめて宣言的に管理し、企業でのガバナンスも見据えたい → Microsoft APM
  • Skillの流通量・コミュニティの厚みを最優先したい → Vercel Skills(skills.sh)
  • 「Agent Package Manager」的な名前のサービスを見つけたが実態が分からない → GitHub star数・コミット履歴・ライセンスの有無・開発元の透明性を確認するまで導入を保留する
