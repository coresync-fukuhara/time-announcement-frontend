Concept 01 / 09

なぜ管理ツールが必要か

説明

Claude Code のようなコーディングエージェントは、そのままでも便利だが、 MCP サーバー(外部システムと繋がるツール)や Skill(特定作業の手順書、/nameで呼び出せる)を追加することで能力を拡張できる。 本プロジェクトでもすでに Playwright MCP・Next.js DevTools MCP を追加した (MCP サーバー導入設計書)。

追加するパーツが増えると、次のような課題が出てくる。

  • 発見: どんな MCP サーバー/Skill が存在するか、公式かどうかをどう調べるか
  • インストール: クライアント(Claude Code, Cursor, VS Code等)ごとに設定形式が違う
  • 更新: バージョンが上がったときに追従する仕組みがあるか
  • 共有: チームメンバーに同じ構成を配れるか
  • 再現性: 自分の環境で動いた構成を、他の開発者やCI環境でも寸分違わず再現できるか
  • セキュリティ: 得体の知れないMCPサーバーや取り込んだSkillを野放しに実行して大丈夫か
  • ガバナンス: チームや組織で「使ってよいもの」を統制できるか

2つの軸: 「発見・インストール」 vs 「宣言的依存管理・再現性」

これらの課題を解決するツールを調べると、大きく2つの方向性に分かれることが分かる。

  • 発見・インストール寄り: 「どんなMCP/Skillがあるか探して、設定に追記する」ことに力点。 レジストリ・マーケットプレイスの充実度や、隔離実行などのセキュリティ機構が差別化要因になる
  • 宣言的依存管理・再現性寄り: npm/pip のように、必要な構成を1ファイルに宣言し (package.json相当)、lockファイルで固定して(package-lock.json相当)、 誰の環境でも寸分違わず再現することに力点。組織的なガバナンス機構を持つものもある

例えるなら、前者は「App Store から都度アプリを探してインストールする」感覚、 後者は「package.json に依存を書いて npm install する」感覚に近い。 パーツの数が少なく、個人利用がメインなら前者で十分なことが多いが、 複数人・複数環境で「寸分違わず同じ構成」を求められるようになると、後者が効いてくる。

「package manager まで必要か」を判断する目安

以下のような状況に当てはまるほど、単なる発見・インストールツールでは足りず、 宣言的依存管理(Microsoft APM 等)まで踏み込む価値が出てくる。逆にどれも当てはまらないなら、 より手軽な発見・インストール系のツールで十分なことが多い。

  • 複数人の開発者が関わり、「私の環境では動くのに」を防ぎたい
  • CI環境やオンボーディング時に、エージェント構成を自動で再現したい
  • 半年後にcloneしても同じ挙動を保証したい(lockファイルによるバージョン固定が欲しい)
  • 組織として「使ってよいSkill/MCPサーバーの出所」を統制したい(ガバナンス要件がある)
  • 複数クライアント(Claude Code・Cursor・Copilot等)を横断して同じ構成を配りたい

本教材の後半(06〜09)では、この「宣言的依存管理・再現性」寄りのツールを深掘りする。

確認問題

  1. MCP/Skillが増えたときに生じる課題を3つ挙げてください。
  2. 「発見・インストール寄り」のツールと「宣言的依存管理・再現性寄り」のツールの違いを、 npmの比喩を使って説明してください。
  3. 自分のプロジェクトで package manager 級のツールが必要かどうかを判断する目安を2つ挙げてください。
