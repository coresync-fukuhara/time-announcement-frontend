Concept 01 / 06

なぜ管理ツールが必要か

説明

Claude Code のようなコーディングエージェントは、そのままでも便利だが、 MCP サーバー(外部システムと繋がるツール)や Skill(特定作業の手順書、/nameで呼び出せる)を追加することで能力を拡張できる。 本プロジェクトでもすでに Playwright MCP・Next.js DevTools MCP を追加した ([MCP サーバー導入設計書](../../../mcp-servers-design.md))。

追加するパーツが増えると、次のような課題が出てくる。

  • 発見: どんな MCP サーバー/Skill が存在するか、公式かどうかをどう調べるか
  • インストール: クライアント(Claude Code, Cursor, VS Code等)ごとに設定形式が違う
  • 更新: バージョンが上がったときに追従する仕組みがあるか
  • 共有: チームメンバーに同じ構成を配れるか
  • セキュリティ: 得体の知れないMCPサーバーを野放しに実行して大丈夫か

例えるなら、スマホにアプリを1つずつ apk ファイルで手動インストールするか、 App Store のようなストアを使うか、という違い。パーツの数が少なければ手動で十分だが、 増えてくると「ストア(管理ツール)」があったほうが楽になる。

今回比較する4つの立ち位置

ツール管理対象提供形態
Claude Code Plugins & MarketplaceSkill + Agent + Hook + MCP + LSP をまとめて1パッケージAnthropic公式・Claude Code専用
SmitheryMCP サーバーのみサードパーティのレジストリ+ホスティング
Docker MCP ToolkitMCP サーバーのみDocker公式・コンテナ実行基盤
install-mcpMCP サーバーのみOSSの軽量CLI(設定ファイル編集の自動化のみ)

ポイント

  • Skill も含めてまとめて管理したいなら、選択肢は実質 Claude Code Plugins のみ (他3つはMCPサーバーのみが対象)
  • 「レジストリ(見つける)」「ホスティング(動かす)」「セキュリティ(隔離実行)」 「軽量インストール(設定を書くだけ)」で、ツールごとに力点が異なる
  • 4つは排他ではなく、Claude Code Plugins ✕ Docker MCP Toolkit のように組み合わせることも可能

確認問題

  1. MCP/Skillが増えたときに生じる課題を3つ挙げてください。
  2. 今回の4候補のうち、Skillの管理まで含んでいるのはどれですか?
