Playwright MCP

説明

Playwright MCP(microsoft/playwright-mcp)は、ブラウザ操作を MCP ツールとして エージェントに公開するサーバー。スクリーンショット画像を都度解析するのではなく、 アクセシビリティツリー(見出し・ボタン・リンクなどの構造化データ)を使って ページの状態を把握するため、軽量かつ決定的に操作できる。

例えるなら、目の見えない人が画面を「見る」代わりに、 画面の構造を音声で読み上げる「スクリーンリーダー」を使って操作するイメージ。 エージェントは画像を推測で解釈する必要がなく、要素名で確実にクリック先を指定できる。

本プロジェクトでは E2E テスト(Playwright, [キャッチアップ教材 07](../../nodejs-test-frameworks/concepts/07-playwright-e2e.html))を 書く/デバッグする際に、エージェントが実際にブラウザを操作しながら 「なぜこのテストが失敗するのか」を調査する用途で活用する。

コード例(設定)

# Claude Code CLI で追加する場合
claude mcp add playwright npx @playwright/mcp@latest

// .mcp.json(プロジェクトに配置し、チームで共有する設定)
{
  "mcpServers": {
    "playwright": {
      "command": "npx",
      "args": ["@playwright/mcp@latest"]
    }
  }
}

ポイント

  • 提供されるツール例: ページ遷移(navigate)、クリック・入力、スクリーンショット、 タブ操作、キーボード/マウス操作
  • ビジョンモデル(画像認識)は不要。アクセシビリティツリーで要素を特定するため、 DOM 構造が変わらない限り安定して動作する
  • VS Code / Cursor / Windsurf / Claude Code など主要な MCP クライアントに対応
  • 本プロジェクトでの活用例: 「初期化ダイアログ→曜日タブ切り替え→保存→再読み込み」 のE2Eシナリオが失敗したとき、エージェントが実際にブラウザを操作して原因を切り分ける

確認問題

  1. Playwright MCP がスクリーンショット画像ではなく「アクセシビリティツリー」を使う利点を説明してください。
  2. 本プロジェクトで Playwright MCP をどのような場面で活用しますか?
