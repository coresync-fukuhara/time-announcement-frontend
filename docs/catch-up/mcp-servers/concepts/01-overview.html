Concept 01 / 03

全体像:公式 MCP の判定基準と採用可否

説明

MCP(Model Context Protocol)は、LLM がツールやリソースを呼び出すための標準プロトコル。 フレームワークの開発元自身が MCP サーバーを公開している場合、 そのフレームワークの内部状態(ビルドエラー・ブラウザの状態など)を エージェントが正確に把握できるようになる。

ただし「公式」を名乗るリポジトリが必ずしも開発元管理とは限らない。 例えるなら、ある製品の「公認代理店」を名乗る店が、実際には本社と関係のない個人商店だった、 というようなケースが MCP エコシステムにも存在する。 採用前に organization(GitHub上の所有者)が開発元と一致するか を必ず確認する。

今回の判定結果

フレームワーク候補判定判定理由
Playwright microsoft/playwright-mcp ✅ 採用 Playwright 自体が Microsoft の OSS。organization が一致し、Playwright 公式ドキュメントからもリンクされている
Next.js vercel/next-devtools-mcp ✅ 採用 Next.js の開発元は Vercel。Next.js 16+ の組み込みエンドポイント(/_next/mcp)に接続する公式コネクタ
Vitest vitest-community/mcp ❌ 非採用 organization が vitest-dev(公式)ではなく vitest-community。 README に "Work in progress" と明記され、開発初期段階

コード例(判定に使った確認観点)

# 確認したポイント(疑似コマンド)
1. GitHub organization が製品の開発元と一致するか
   → microsoft/playwright-mcp: 一致(Playwright = Microsoft製)
   → vercel/next-devtools-mcp: 一致(Next.js = Vercel製)
   → vitest-community/mcp: 不一致(Vitest公式は vitest-dev)

2. 公式ドキュメントからリンクされているか
   → Playwright: playwright.dev からリンクあり
   → Next.js: nextjs.org/docs からリンクあり
   → Vitest: vitest.dev からのリンクは確認できず

ポイント

  • 「official」という単語がREADMEにあっても、organization の一致まで確認する
  • 本プロジェクトでは Playwright MCP と Next.js DevTools MCP の2つを採用
  • MSW・Ajv には調査時点(2026年7月)で公式MCPが存在しないため、追加しない
  • Vercel プラットフォーム全体の MCP(デプロイ管理用)は、自前 Docker 運用のため対象外

確認問題

  1. 「公式」を名乗る MCP サーバーを採用する前に、何を確認すべきですか?
  2. 今回 Vitest MCP を採用しなかった理由を2つ挙げてください。
